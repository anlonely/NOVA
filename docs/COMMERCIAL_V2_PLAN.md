# Nebula Interp Console 商用版方案

## 1. 产品定义

Nebula Interp Console 不是一个“翻译按钮”，而是一个面向 Discord、会议、直播、播客和跨语种沟通场景的双通道语音路由台。

目标用户：

- Discord 跨语种语音用户
- 跨境销售 / 客服
- 线上会议主持人
- 直播和播客制作团队
- 需要双语字幕和译音的内容创作者

核心卖点：

- 双通道独立配置
- 低延迟实时译音
- 双语字幕并行
- 对虚拟声卡友好
- 支持音色复刻和固定音色两种策略
- 场景预设和会话导出

## 2. 对标结论

这版方案参考了当前主流产品的公开能力，提炼出几条必须对齐的体验标准：

- Teams：语言通道、原声/译声音量平衡、实时字幕和多语言识别
- Zoom：参与者自己选字幕语言、完整 transcript 侧栏、译字幕对会议流程低侵入
- Interprefy：对现有会议栈和虚拟音频链路兼容，不要求用户重搭 AV
- KUDO：实时译音 + 字幕并行，强调“连续说话、持续翻译”
- Seed LiveInterpret 2.0：产品级语音复刻和 2-3 秒级低延迟是行业标杆方向

## 3. 信息架构

商用版桌面端建议拆成 6 个一级区域：

1. 顶部场景栏
2. 左侧引擎与凭证中心
3. 中央实时路由图
4. 右侧诊断 / 质量看板
5. 底部双通道控制卡片
6. 浮窗字幕层

## 4. 通道配置清单

每个通道独立可配：

- 输入设备
- 输出设备
- 监听设备
- 回环输入
- 源语言
- 目标语言
- 中英混说模式
- 固定音色 ID / 自动复刻
- 性能档位
- 分包时长
- 播放缓冲
- 输出采样率
- 输入增益
- 字幕模式

未来版本扩展：

- 降噪
- VAD
- AGC
- 原声 ducking
- 术语表 / 热词 / 替换词
- 说话人分离
- 自动场景切换

## 5. 延迟目标

建议用档位而不是单一承诺值：

### Turbo

- 原文局部字幕：200-400ms
- 译文局部字幕：450-800ms
- 译音起播：0.8-1.6s

### Balanced

- 原文局部字幕：250-500ms
- 译文局部字幕：600-1000ms
- 译音起播：1.5-2.5s

### Studio

- 原文局部字幕：400-700ms
- 译文局部字幕：900-1400ms
- 译音起播：2.5-4.0s

说明：

- 这是工程目标，不是对任何第三方模型的官方承诺
- 当启用音色复刻并要求更自然的语气时，端到端延迟通常会上升

## 6. 技术架构

### 当前可交付版本

- UI：Python + Tkinter
- 音频采集/播放：soundcard / WASAPI 路径
- 翻译引擎：火山 AST WebSocket
- 协议：官方 protobuf
- 配置：本地 JSON
- 导出：JSON + TXT

### 商用目标架构

- UI：Tauri + React
- 音频核心：Rust
- 实时链路：
  - Input Capture
  - DSP / VAD / Ducking
  - Provider Adapter
  - Subtitle Stream
  - Audio Jitter Buffer
  - Output Mixer
- Provider 抽象：
  - Volc AST Provider
  - Future S2S Provider
  - 本地推理 Provider

## 7. 核心工程原则

- 主音频路径不走普通 UI 队列
- 实时路径与日志路径分离
- 指标和字幕优先于花哨动画
- 所有网络和设备故障都必须可恢复
- 不依赖用户手工重新连线

## 9. 商用交付标准

上线前至少满足：

- 连续运行 2 小时稳定
- 设备切换恢复 < 2 秒
- 网络闪断重连 < 3 秒
- 双通道不互相啸叫
- 导出 transcript 正常
- UI 配置不丢失
- 遇到 API 错误有明确诊断
